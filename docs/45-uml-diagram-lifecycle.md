# UML Diagram Lifecycle Rule (Level 2.3)

**PURPOSE:** Define which UML diagrams must be generated, when generation triggers, where outputs go, and what format to use. This rule drives `langgraph_engine/diagrams/` and the draw.io generator — scripts execute, but this rule decides what.

---

## 1. Standard Diagram Set

Every project must maintain all 13 diagram types. Generate ALL types on first run; on subsequent runs, regenerate only the types affected by the current change (see §3).

| # | Diagram Type | File Name | When Affected |
|---|-------------|-----------|---------------|
| 1 | Class Diagram | `class_diagram.{ext}` | Any class/interface added, modified, removed |
| 2 | Package Diagram | `package_diagram.{ext}` | Package/module structure changed |
| 3 | Component Diagram | `component_diagram.{ext}` | New service, library, or integration added |
| 4 | Sequence Diagram | `sequence_diagram.{ext}` | Method call chains changed |
| 5 | State Diagram | `state_diagram.{ext}` | State machine or workflow logic changed |
| 6 | Activity Diagram | `activity_diagram.{ext}` | Business logic or process flow changed |
| 7 | Deployment Diagram | `deployment_diagram.{ext}` | Infrastructure or Docker/K8s config changed |
| 8 | Use Case Diagram | `usecase_diagram.{ext}` | New user-facing feature added |
| 9 | Object Diagram | `object_diagram.{ext}` | Key object relationships changed |
| 10 | Composite Structure | `composite_diagram.{ext}` | Internal structure of complex class changed |
| 11 | Interaction Overview | `interaction_diagram.{ext}` | Interaction between subsystems changed |
| 12 | Communication Diagram | `communication_diagram.{ext}` | Message/event flow between components changed |
| 13 | Call Graph | `call_graph_diagram.{ext}` | Any function/method added or removed |

---

## 2. Output Locations

| Format | Directory | Environment Variable Override |
|--------|-----------|-------------------------------|
| Mermaid (`.md`) | `{project_root}/uml/` | `UML_OUTPUT_DIR` |
| Draw.io (`.drawio`) | `{project_root}/drawio/` | `DRAWIO_OUTPUT_DIR` |

Rules:
- Both directories are at the **project root**, not inside `docs/`
- Create directories if they do not exist
- Never commit generated diagram files to Git (add both dirs to `.gitignore`)
- Overwrite previous version of same diagram type — do NOT version diagram files

---

## 3. Generation Triggers

### 3.1 Full Regeneration (all 13 types)

Trigger full regeneration when:
- Step 13 runs for the first time (no `uml/` directory exists)
- A major architectural change occurred (new pipeline level, new package, new service)
- `FORCE_DIAGRAM_REBUILD=1` environment variable is set

### 3.2 Selective Regeneration (affected types only)

Trigger selective regeneration at Step 13 for the types listed in §1 whose "When Affected" condition matches the current task's file changes.

Detection: compare the diff from Step 10 (implementation) against the condition column in §1 table.

### 3.3 Skip Diagram Generation

Skip entirely when the task type is:
- `chore`, `test`, `docs` (no structural code change)
- The task only modified files in `tests/`, `docs/`, `uml/`, `drawio/`

## 4. Format Rules

### 4.1 Mermaid Format (uml/ directory)

- One `.md` file per diagram type
- File begins with a frontmatter comment: `<!-- Generated by pipeline Step 13 — do not edit manually -->`
- Diagram wrapped in Mermaid fenced block: ` ```mermaid `
- Use Mermaid syntax compatible with GitHub rendering (no Kroki-only extensions)
- Max node count per diagram: 50 nodes. If exceeding, show top-level only and add note: `%% Truncated: showing top {N} nodes`

### 4.2 Draw.io Format (drawio/ directory)

- One `.drawio` file per diagram type (XML format)
- Editable in draw.io desktop, app.diagrams.net, and VS Code draw.io extension
- Include a shareable URL comment at top of file if `DRAWIO_SHARE=1` is set
- Do NOT embed binary assets — keep files text-only XML

  interaction_diagram.md
  communication_diagram.md
  call_graph_diagram.md

drawio/
  class_diagram.drawio
  package_diagram.drawio
  ... (same stem, .drawio extension)
```

Use exactly these names — no timestamps, no version suffixes, no camelCase.

## 6. Data Source Priority

When generating diagrams, use data sources in this priority order:

1. **CallGraph from `call_graph_builder.py`** — primary source (AST-parsed, 4 languages)
2. **Pre-analysis result** from `state["pre_analysis_result"]` — enrichment data
3. **AST direct scan** via `diagrams/ast_analyzer.py` — fallback if CallGraph unavailable
4. **LLM enrichment** — only for Use Case and Activity diagrams where intent matters

Never use LLM as primary source for structural diagrams (class, package, component, sequence).

---

## 7. Failure Handling

| Failure | Action |
|---------|--------|
| CallGraph build fails | Fall back to AST direct scan; log WARNING |
| AST scan fails on a file | Skip that file; log WARNING with filename |
| Mermaid render invalid | Log ERROR with diagram type; skip that type; do not block Step 13 |
| Draw.io XML invalid | Log ERROR; skip draw.io for that type; Mermaid version still saved |
| uml/ directory not writable | Log ERROR; block Step 13 completion |

Diagram generation failures MUST NOT block the SDLC pipeline. Log and continue.

---

## 8. What This Rule Replaces

This rule replaces hardcoded logic in:
- `langgraph_engine/diagrams/legacy_generator.py` → diagram type list and trigger logic
- `langgraph_engine/level3_execution/documentation_manager.py` → `_generate_drawio_diagrams()`
- `langgraph_engine/uml_generators.py` → which types to generate

Scripts read this rule to decide WHICH diagrams to generate and WHEN. Scripts decide HOW to generate them.
