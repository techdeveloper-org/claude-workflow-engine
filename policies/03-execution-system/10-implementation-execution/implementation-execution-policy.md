# Implementation Execution Policy - Step 10

**Version:** 1.0.0

**Step:** 10 - Implementation Execution
**Status:** Active

## Overview

Step 10 executes the actual implementation using the system prompt and user message generated by Step 7. It invokes hybrid LLM inference with full context (system prompt) to achieve 95%+ execution quality.

**Input:** `system_prompt.txt` + `user_message.txt` from Step 7
**Output:** LLM response with implementation guidance, modified files list

---

## Policy Scope

**Applies to:** Level 3: Execution System, Step 10
**Predecessor:** Step 9 (Branch Creation + Git Setup)
**Successor:** Step 11 (Pull Request & Code Review)
**RAG Threshold:** Not applicable (implementation is unique per task)
**Standards Hook:** `apply_integration_step10` - Code review compliance checklist

## Execution Modes

### Hook Mode (CLAUDE_HOOK_MODE=1) - Default

In hook mode, Step 10 is **NOT executed by the pipeline**. Instead:
1. Steps 0-7 prepare context (analysis, prompt generation)
2. Steps 8-9 create GitHub resources (issue, branch)
3. Pipeline outputs synthesized prompt and STOPS
4. **Claude Code itself** performs the implementation using the generated prompt
5. Stop hook auto-triggers Steps 11-14 (PR, closure, docs, summary)

### Full Mode (CLAUDE_HOOK_MODE=0)

In full mode, Step 10 executes within the pipeline:
1. Reads `system_prompt.txt` from session folder (Step 7 output)
2. Reads `user_message.txt` from session folder (Step 7 output)
3. Invokes hybrid inference with system prompt as context
4. Captures LLM response and tracks implementation status

---

## System Prompt Integration

### Mandatory: System Prompt Required

Step 10 uses a **fail-fast** approach:
- If `system_prompt.txt` is missing → Step 10 returns `ERROR` immediately
- Rationale: Without system prompt context, execution quality drops from 95% to 60%
- Step 7 (Final Prompt Generation) MUST succeed before Step 10 can run

### System Prompt Contains (from Step 7)

- Complete task breakdown (from Steps 0-3)
- Full skill/agent definitions (not truncated)
- Execution plan (from Step 2, if applicable)
- Project information and standards
- TOON context (from Step 4)

### User Message Fallback

If `user_message.txt` is missing but system prompt exists:
- Build basic message: `"Execute the {task_type} based on the breakdown and resources provided."`
- This is acceptable because system prompt provides full context

---

## LLM Invocation

### Hybrid Inference

Step 10 uses the hybrid inference manager:
1. **GPU-first:** Try local Ollama model
2. **Cloud fallback:** Fall back to Claude API if local unavailable
3. Step name: `step10_implementation_execution`
4. System prompt passed separately for full context injection

### Response Handling

| Condition | Action |
|-----------|--------|
| LLM returns `status: "ok"` | Capture response, mark `llm_invoked: True` |
| LLM returns error | Log warning, store error message |
| LLM unavailable | Mark `llm_invoked: False`, continue pipeline |

---

## Retry Loop (Step 11 Integration)

Step 10 participates in a retry loop with Step 11:
1. Step 10 executes implementation
2. Step 11 reviews the PR
3. If review fails AND retry_count < 3 → Re-execute Step 10
4. Retry context includes previous review feedback

### Retry Context

On retry, Step 10 receives:
- `step11_review_issues` - Issues found in previous review
- `step11_retry_count` - Current retry attempt number
- Previous implementation response for comparison

## Output State Keys

| Key | Type | Description |
|-----|------|-------------|
| `step10_implementation_status` | str | `OK`, `ERROR`, or `SKIPPED` |
| `step10_system_prompt_loaded` | bool | Whether system prompt was found |
| `step10_user_message_loaded` | bool | Whether user message was found |
| `step10_llm_invoked` | bool | Whether LLM was actually called |
| `step10_llm_response` | str | Raw LLM response text |
| `step10_tasks_executed` | int | Number of tasks processed |
| `step10_modified_files` | list | Files modified during implementation |
| `step10_error` | str | Error message (if status is ERROR) |

---

## Quality Requirements

1. **System prompt is MANDATORY** - Never execute without full context
2. **Track all outputs** - Modified files list must be accurate for Steps 11-14
3. **Respect retry feedback** - On retry, incorporate Step 11 review comments
4. **Log inference details** - Response length, provider used, latency

---

## Error Handling

- Missing system prompt: `ERROR` status, pipeline continues to Step 11 (which will also fail gracefully)
- LLM timeout: Log warning, mark as not invoked
- LLM error: Capture error message in response field
- Exception: Return `ERROR` status with details
