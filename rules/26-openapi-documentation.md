# OpenAPI Documentation (Level 2.2 - Spring Boot)

**PURPOSE:** Enforce consistent OpenAPI 3.x annotations on every controller so that
Swagger UI generated by the gateway presents a complete, accurate, and navigable API
contract without gaps or undocumented endpoints.

**APPLIES WHEN:** Spring Boot project with `springdoc-openapi-starter-webmvc-ui` dependency.

---

## 1. Controller-Level @Tag

### What We Follow
Every `@RestController` carries `@Tag(name=..., description=...)`. The `name` matches
the entity/resource domain. The `description` explains what the controller manages.

### How To Implement

```java
// CORRECT -- controller tagged at class level
@RestController
@RequestMapping(ApiConstants.{ENTITY}_BASE)
@RequiredArgsConstructor
@Tag(
    name = "{Entity} Management",
    description = "APIs for creating, retrieving, updating, and deleting {Entity} resources"
)
public class {Entity}Controller { ... }

// WRONG -- no @Tag; endpoint appears under default "default" group in Swagger UI
@RestController
@RequestMapping(ApiConstants.{ENTITY}_BASE)
public class {Entity}Controller { ... }
```

### Why This Matters
Without `@Tag`, all undocumented controllers collapse into a single "default" group in
Swagger UI, making the API contract impossible to navigate for new developers.

## 2. @Operation on Every Endpoint Method

### What We Follow
Every `@GetMapping`, `@PostMapping`, `@PutMapping`, `@PatchMapping`, and `@DeleteMapping`
method carries `@Operation(summary=..., description=...)`.

### How To Implement

```java
// CORRECT -- fully documented endpoint
@GetMapping(ApiConstants.BY_ID)
@Operation(
    summary = "Get {Entity} by ID",
    description = "Retrieves a single {Entity} by its unique identifier. "
                + "Returns 404 if the {Entity} does not exist."
)
@ApiResponses(value = {
    @ApiResponse(responseCode = "200", description = "Successfully retrieved {Entity}",
        content = @Content(schema = @Schema(implementation = ApiResponseDtoOf{Entity}Dto.class))),
    @ApiResponse(responseCode = "404", description = "{Entity} not found",
        content = @Content(schema = @Schema(implementation = ApiResponseDtoVoid.class))),
    @ApiResponse(responseCode = "500", description = "Internal server error",
        content = @Content(schema = @Schema(implementation = ApiResponseDtoVoid.class)))
})
public ResponseEntity<ApiResponseDto<{Entity}Dto>> get{Entity}ById(
        @Parameter(description = "Unique identifier of the {Entity}", required = true)
        @PathVariable Long id) {
    ApiResponseDto<{Entity}Dto> response = {entity}Services.get{Entity}ById(id);
    return ResponseEntity.status(HttpStatus.valueOf(response.getStatus())).body(response);
}

// WRONG -- no documentation
@GetMapping("/{id}")
public ResponseEntity<ApiResponseDto<{Entity}Dto>> get{Entity}ById(@PathVariable Long id) {
    // Appears in Swagger with no summary, no response codes, no parameter description
}
```

Standard response codes to document per method type:
- GET: 200, 400, 404, 500
- POST: 201, 400, 409, 500
- PUT/PATCH: 200, 400, 404, 409, 500
- DELETE: 200, 404, 500

### Why This Matters
Undocumented response codes mean consumers cannot write correct error-handling code.
They discover 409 Conflict responses in production rather than from the API contract.

## 3. @Parameter on Path and Query Variables

### What We Follow
Every `@PathVariable` and `@RequestParam` parameter carries `@Parameter(description=..., required=...)`.

### How To Implement

```java
// CORRECT -- parameters documented
@GetMapping
@Operation(summary = "List {Entity}", description = "Returns paginated list of {Entity} records")
public ResponseEntity<ApiResponseDto<Page<{Entity}Dto>>> list{Entity}(

        @Parameter(description = "Zero-based page number", required = false)
        @RequestParam(defaultValue = "0") int page,

        @Parameter(description = "Number of records per page (max 100)", required = false)
        @RequestParam(defaultValue = "20") int size,

        @Parameter(description = "Filter by status", required = false)
        @RequestParam(required = false) {Entity}Status status) {
    // ...
}

// WRONG -- no @Parameter; Swagger shows "string" type with no description
@GetMapping
public ResponseEntity<ApiResponseDto<Page<{Entity}Dto>>> list{Entity}(
        @RequestParam(defaultValue = "0") int page,
        @RequestParam(defaultValue = "20") int size) { ... }
```

### Why This Matters
Without `@Parameter`, Swagger UI shows query parameters as unlabeled input boxes with no
hint about valid values, ranges, or whether they are required.

## 4. JWT Security Scheme in OpenAPI Config

### What We Follow
A shared `OpenApiConfig` in common-lib defines the JWT bearer token security scheme.
Per-service OpenAPI beans add the scheme to all their operations.

### How To Implement

```java
// CORRECT -- OpenApiConfig in common-lib (shared)
@Bean
public OpenAPI openApiDefinition() {
    return new OpenAPI()
        .addSecurityItem(new SecurityRequirement().addList("bearerAuth"))
        .components(new Components()
            .addSecuritySchemes("bearerAuth",
                new SecurityScheme()
                    .name("bearerAuth")
                    .type(SecurityScheme.Type.HTTP)
                    .scheme("bearer")
                    .bearerFormat("JWT")
            )
        )
        .info(new Info()
            .title("{service} API")
            .description("REST API for {service}")
            .version("1.0.0")
        );
}

// WRONG -- no security scheme; Swagger UI has no way to add Authorization header
// Developer must manually set the header in Postman for every test
```

### Why This Matters
Without the security scheme, developers cannot authenticate through Swagger UI's "Try it out"
feature, forcing them to switch to external tools for every API test.
